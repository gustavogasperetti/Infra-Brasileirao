### 3. `etl/load.py` — Camada Load (Gold → PostgreSQL) — v1.0.0

Implementação completa do script de ingestão dos CSVs Gold no banco de dados PostgreSQL.

**Funcionalidades:**

- **Leitura de credenciais via `.env`** (reutiliza engine do `api/database.py`)
- **Estratégia idempotente (DROP + CREATE + INSERT):** Cada execução recria as tabelas do zero, garantindo que os dados sejam sempre consistentes com o CSV Gold.
- **Carga via `pandas.to_sql()`** com `chunksize=500` e `method="multi"` para inserções otimizadas em batch.
- **Validação pós-carga:**
  - Contagem de registros em ambas as tabelas
  - Verificação de integridade referencial (FKs órfãs)
- **Logging estruturado** com tempo de execução total

**Fluxo de execução:**
1. Verifica se os CSVs Gold existem
2. Carrega `dim_times.csv` e `fato_partidas_ouro.csv` em DataFrames
3. Remove tabelas existentes (DROP ALL)
4. Cria tabelas via DDL do SQLAlchemy (CREATE ALL)
5. Insere dimensão primeiro (por causa das FKs)
6. Insere tabela fato
7. Valida contagens e integridade referencial

**Resultado da execução:**

| Tabela | Registros | Tempo |
|--------|-----------|-------|
| `dim_times` | 167 | 0.1s |
| `fato_partidas_ouro` | 21.453 | 15.5s |
| **Total** | **21.620** | **15.7s** |
| FKs órfãs | 0 | ✅ |

**Uso:** `python -m etl.load`

## 🐛 Bugs Encontrados e Resolvidos

### Bug 1 — `psycopg2` UnicodeDecodeError no Windows

**Erro:** `UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe7 in position 78: invalid continuation byte`

**Causa:** O driver `psycopg2-binary` utiliza a biblioteca C `libpq` para se conectar ao PostgreSQL. No Windows com locale `pt_BR/CP1252`, a libpq lê informações do sistema que contêm o caractere `ç` (byte `0xe7` em CP1252) e tenta decodificar como UTF-8, falhando.

**Solução:** Migração do driver `psycopg2-binary` para **`psycopg[binary]` (v3)**, que é implementado em Python puro e não sofre deste bug. O dialect do SQLAlchemy foi alterado de `postgresql+psycopg2` para `postgresql+psycopg`. O `requirements.txt` foi atualizado.

---

### Bug 2 — Conflito de Porta 5432

**Erro:** `FATAL: autenticação do tipo senha falhou para o usuário "brasileirao"`

**Causa:** Havia um PostgreSQL local já instalado no Windows escutando na porta 5432 (PID 8732). O Docker também mapeava para a porta 5432, mas as conexões Python iam para o PostgreSQL local (que tinha credenciais diferentes), gerando falha de autenticação. Identificado via `netstat -ano | findstr ":5432"` (dois processos na mesma porta).

**Solução:** Remapeamento da porta do container Docker de `5432` para **`5433`**. Alterado em `docker-compose.yml` e `.env` (`DB_PORT=5433`).

---

### Bug 3 — Limite de Parâmetros do PostgreSQL

**Erro:** `psycopg.OperationalError: number of parameters must be between 0 and 65535`

**Causa:** O `pandas.to_sql()` com `method="multi"` e `chunksize=5000` gerava `66 colunas × 5000 linhas = 330.000 parâmetros` por query, excedendo o limite de 65.535 parâmetros por statement do PostgreSQL.

**Solução:** Redução do `chunksize` de 5000 para **500** (`66 × 500 = 33.000 < 65.535`).

## 📝 Observações e Decisões Técnicas

- **Estratégia de carga idempotente:** O `load.py` usa DROP + CREATE a cada execução. Isso é intencional — os dados são regeneráveis via pipeline Gold, e a recriação garante consistência total.
- **Pool de conexões:** Engine configurado com `pool_size=10` e `max_overflow=20` para suportar múltiplas requisições concorrentes na API.
- **CORS aberto:** Para o contexto de TCC, o CORS está liberado para todas as origens. Em produção, seria restrito aos domínios do front-end.
- **Prefixo `/api/v1`:** Todos os endpoints usam versionamento na URL para facilitar futuras breaking changes.
- **Serialização manual:** Optou-se por serialização via `_serializar_partida()` ao invés de Pydantic `response_model` para evitar definir schemas de 66 campos. O Swagger ainda documenta via docstrings.
- **Migração psycopg2 → psycopg v3:** Necessária por bug de encoding da libpq C no Windows com locale `pt_BR`. O psycopg v3 é o driver oficial recomendado atualmente e é compatível com SQLAlchemy 2.x via dialect `postgresql+psycopg`.
- **Porta 5433:** Escolhida para evitar conflito com PostgreSQL local. Documentada no `.env` e no `docker-compose.yml`. A porta interna do container continua sendo 5432 (padrão).
- **Chunksize 500:** Ajustado para respeitar o limite de 65.535 parâmetros por statement do PostgreSQL ao usar `method="multi"` com 66 colunas.

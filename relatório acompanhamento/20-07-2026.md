## 🎯 Resumo da Sessão

Refatoração completa da infraestrutura: o projeto abandonou o PostgreSQL + FastAPI e passou a usar **Google Sheets como repositório final de dados**, alimentado por uma pipeline **serverless no GitHub Actions** (seg/qua/sex). A camada Ouro migrou de fato/dimensão para **One Big Table (OBT)**, e a fonte de extração corrente migrou do **Ogol para o frontend da CBF** após diagnóstico de bloqueio anti-bot no CI.

## ✅ O que foi feito nesta data

### 1. Desacoplamento de Infraestrutura (remoção do banco de dados)

- **Removidos:** `docker-compose.yml`, pasta `api/` inteira (FastAPI, SQLAlchemy, models, routes) e as dependências `SQLAlchemy`, `psycopg`, `fastapi`, `uvicorn` do `requirements.txt`.
- **Adicionados:** `gspread` e `google-auth` para a comunicação com o Google Sheets.
- **Segurança:** todas as credenciais saíram do código. Contas do Ogol (que estavam hardcoded com senha no `config.py`) e credenciais Google agora vêm de variáveis de ambiente alimentadas por Secrets no CI e por `.env` local (criado `.env.example` documentando o formato).
- **Decisão estrutural:** a pasta `data/` (bronze/silver/gold) passou a ser **versionada no git** — o runner do CI nasce de um checkout limpo e precisa do histórico 1971–2025 para reconstruir a OBT completa re-extraindo apenas o ano corrente.

### 2. `etl/gold.py` v3.0.0 — One Big Table

- Abandono do modelo fato/dimensão: saída única `data/gold/brasileirao_obt.csv` (21 colunas), sem `dim_times`, sem FKs — estados (UF) desnormalizados na própria linha.
- **Removidos os cálculos temporais** (H2H, forma dos últimos 5 jogos, sequências, acumulados, fadiga/dias de descanso, clean sheets): a responsabilidade é da futura biblioteca `brasileirao-lib`. A pipeline entrega apenas a "fotografia" de cada partida.
- Mantidos os enriquecimentos por partida (não temporais): `total_gols`, saldos, pontos com regra histórica (2 pts < 1995, 3 pts ≥ 1995), `tipo_fase`/`is_mata_mata`, `is_classico_estadual`.
- Dicionário de dados reescrito para as 21 colunas.

### 3. `etl/load.py` v2.0.0 — Publicação no Google Sheets

- Autenticação por Service Account com prioridade: `GOOGLE_CREDENTIALS` (env/Secret) → `GOOGLE_APPLICATION_CREDENTIALS` → `credentials.json` local.
- **Modo `overwrite`:** limpa e regrava a aba (idempotente), em chunks de 5.000 linhas.
- **Modo `append` = upsert incremental:** compara OBT × planilha pela **chave natural** (`ano_campeonato + Mandante + Visitante + Fase`), atualiza in-place linhas modificadas (jogo futuro que ganhou placar, adiamento, correção) via `values.batchUpdate` e acrescenta apenas partidas inéditas. Fallbacks automáticos para regravação completa: aba vazia, cabeçalho divergente ou > 2.000 modificações.
- A lógica de comparação é função pura (`calcular_diff`), testável sem tocar na API.

### 4. `.github/workflows/etl.yml` — Orquestração no GitHub Actions

- Agendamento **seg/qua/sex às 06:00 UTC (03:00 BRT)** + disparo manual com input de anos.
- Fluxo: Extract (ano corrente) → Transform → Gold → Load → **commit automático** dos CSVs atualizados.
- Configuração com precedência: input manual > Secret > Variable > padrão (`ANOS_EXTRACAO`, `LOAD_MODO`).
- Em falha, a pasta `debug/` (screenshots + HTML do navegador) é publicada como artifact `debug-playwright`.

### 5. Migração de fonte: Ogol → frontend da CBF

- **Diagnóstico:** primeira execução no Actions falhou no login do Ogol. O artifact de debug revelou **Cloudflare Turnstile** ("Confirme que é humano") — o Ogol desafia IPs de datacenter; da máquina local (IP residencial BR) tudo funciona. Estratégias intermediárias (navegação anônima com login como fallback, fechamento de banners de consentimento) foram implementadas, mas o desafio bloqueia qualquer página.
- **Sondagem de fontes alternativas** (headers/assinaturas anti-bot + workflow de probe rodando de dentro do runner): CBF, Wikipedia e ge.globo livres; football-data.org e api-futebol como APIs com chave.
- **Escolha: CBF** (`/futebol-brasileiro/tabelas/campeonato-brasileiro/serie-a/{ano}`), por decisão de scraping no frontend oficial.

### 6. `etl/extract.py` v7.0.0 — Scraping do frontend da CBF

- Playwright abre a página do ano, aguarda os cards de jogos substituírem os skeletons do React, itera o **seletor de rodadas** (1..38) e o BeautifulSoup parseia cada card.
- Nome completo do clube extraído do atributo `title` do `<strong>` (o texto visível é abreviado); gols nos spans; data no bloco de informações. Jogos futuros → placar vazio.
- **Seletores por fragmento estável de classe** (`div[class*='gameCardContainer']`) — resistem aos sufixos hasheados que o Next.js regenera a cada deploy.
- Robustez: dump de screenshot+HTML em `debug/` para rodada sem cards ou falha de ano; `exit 1` se nenhum ano for extraído (impede o Load de publicar dados desatualizados); `ANOS_EXTRACAO` (`atual`/`todos`/lista, cobertura ≥ 2012).
- **Extrator do Ogol preservado** como `etl/extract_ogol.py` (proveniência/reprodutibilidade do histórico 1971–2011, fora da cobertura da CBF).
- De-para de times ampliado com as grafias da CBF: `Santos FC`→`Santos`, `Coritiba SAF`→`Coritiba`, `Vasco da Gama Saf`→`Vasco` (os outros 17 clubes de 2026 já batiam).

## 🐛 Bugs e Problemas Resolvidos

### Bug 1 — Workflow ignorava os Secrets de configuração
**Sintoma:** com `ANOS_EXTRACAO=2026` e `LOAD_MODO=append` cadastrados como Secrets, o run usou `atual`/`overwrite`.
**Causa:** os valores estavam hardcoded no YAML — Secrets não entram no ambiente sozinhos; cada step precisa referenciá-los.
**Solução:** precedência explícita `input manual > secrets.X > vars.X > padrão` nos dois envs.

### Bug 2 — Extração falhava e a pipeline continuava
**Sintoma:** login do Ogol falhou no CI e mesmo assim Transform/Gold/Load rodaram, sobrescrevendo a planilha com dados de 1971–2025 (sem o ano corrente).
**Causa:** `extract.py` logava o erro mas retornava exit code 0.
**Solução:** `sys.exit(1)` quando nenhum ano é extraído — o job para antes do Load.

### Bug 3 — Modo `append` quebrava com jogos futuros
**Sintoma:** como a OBT publica o calendário completo (jogos futuros com placar vazio), o append incremental por `id_partida` nunca atualizaria os placares (o id já existia na planilha); além disso, `id_partida` se desloca quando um jogo é adiado (é regenerado por ordenação cronológica).
**Solução:** upsert pela chave natural `ano + Mandante + Visitante + Fase` (estável a adiamentos), com update in-place das linhas modificadas.

### Bug 4 — Gols subiam como float ("1.0") para a planilha
**Causa:** `read_csv` promove colunas Int64 com nulos para float.
**Solução:** restauração de `Int64` nullable pós-leitura para colunas float integrais.

### Bug 5 — Bloqueio Cloudflare Turnstile no Ogol (IPs de datacenter)
**Diagnóstico:** artifact de debug do runner mostrou a tela "Executando verificação de segurança" em todas as páginas. Não é contornável por código — motivou a troca de fonte para a CBF (item 5).

### Bug 6 — SSL em rede corporativa (desenvolvimento local)
**Sintoma:** `SSLError: certificate verify failed` no `requests` local, enquanto `curl` funcionava.
**Causa:** inspeção TLS corporativa — o `certifi` do Python não conhece a CA da rede; o `curl` usa o repositório de certificados do Windows.
**Observação:** irrelevante no CI; a extração final via Playwright não é afetada.

## 📝 Observações e Decisões Técnicas

- **Serverless por design:** custo zero de infraestrutura — GitHub Actions executa, o repositório versiona os dados e o Google Sheets serve o consumo.
- **Dados versionados:** sem o histórico no git, o runner (checkout limpo) perderia 1971–2025 a cada execução. Os CSVs são o ativo do projeto open-source.
- **Overwrite vs Append:** `overwrite` continua o padrão (idempotente e barato para ~21,8 mil linhas); `append` virou upsert de verdade para minimizar escrita quando desejado.
- **Horário do cron (03:00 BRT):** todos os jogos da noite anterior já terminaram; o run de segunda captura o fim de semana, o de sexta captura as rodadas de meio de semana.
- **Cron do GitHub:** disparo pode atrasar em horários de pico; agendamentos pausam após ~60 dias sem atividade — auto-resolvido pelos commits automáticos da própria pipeline.
- **Fragilidade do scraping de frontend:** mais lento que endpoint (~2 min vs ~30 s) e sensível a redesigns da CBF — mitigado com seletores por fragmento de classe e dumps de debug publicados como artifact para correção rápida.
